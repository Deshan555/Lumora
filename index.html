01

How the app works — end to end

The core idea is straightforward: instead of sending your message to a remote server and waiting for a response, BRAINY.AI loads a language model into your phone's RAM and runs inference locally. The result is streamed token-by-token directly to the UI — no network latency, no API keys, no data leaving the device.

llama.cpp generates tokens using the KV cache and selected sampler settings (temperature, top-p, top-k). Each token fires a Dart callback that appends to the streaming message buffer in the UI. The first token typically arrives in 500ms–2s depending on device tier.

Code architecture — how it's structured

The app follows a clean domain-driven architecture. The presentation layer never touches data sources directly — it goes through providers, which call services and repositories, which sit on top of the data layer. This keeps the codebase testable and the inference engine completely swappable.

┌─────────────────────────────────────────────────────┐ │ PRESENTATION │ │ Screens · Widgets · go_router · Riverpod providers │ └────────────────────┬────────────────────────────────┘ │ watches / reads ┌────────────────────▼────────────────────────────────┐ │ DOMAIN │ │ ModelInfo · AppSettings · AIPersonality │ │ Repositories (interfaces only — no impl here) │ └──────────────┬──────────────────────┬───────────────┘ │ implements │ calls ┌──────────────▼──────┐ ┌───────────▼──────────────┐ │ DATA SOURCES │ │ SERVICES │ │ LLMService │ │ TTSService │ │ HuggingFaceService │ │ VoiceInputService │ │ LlamaCppEngine │ │ NotificationService │ │ DownloadManager │ │ SystemMonitorService │ └──────────────┬──────┘ └───────────────────────────┘ │ reads/writes ┌──────────────▼──────────────────────────────────────┐ │ PERSISTENCE │ │ Drift/SQLite · SharedPreferences · SecureStorage │ │ /proc/meminfo · /proc/stat · GGUF files on disk │ └─────────────────────────────────────────────────────┘

LLMService — the orchestration core

LLMService is the master orchestrator. It owns the inference lifecycle — loading, prompting, streaming, and unloading. All other parts of the app talk to this service, never to llama.cpp directly.

Dart LLMService — simplified inference flow 
class LLMService {
  LlamaCppEngine? _engine;
  ModelRuntime _runtime = ModelRuntime.auto;

  // Detects best GPU backend for the loaded model
  Future<ModelRuntime> detectRuntime(ModelInfo model) async {
    final backends = await VulkanDetector.getAvailable();
    if (backends.contains('vulkan')) return ModelRuntime.llamaCppVulkan;
    if (backends.contains('opencl')) return ModelRuntime.llamaCppOpenCL;
    return ModelRuntime.llamaCpp; // CPU fallback
  }

  // Streams tokens to the caller via callback
  Future<void> generateStream({
    required String prompt,
    required AppSettings settings,
    required Function(String) onToken,
    Function? onComplete,
  }) async {
    final formatted = PromptTemplateService.format(
      prompt, _activeModel!.architecture
    );
    await _engine!.generate(
      prompt: formatted,
      temperature: settings.temperature,
      topP: settings.topP,
      topK: settings.topK,
      maxTokens: settings.maxTokens,
      tokenCallback: onToken,
    );
    onComplete?.call();
  }
}

State management with Riverpod

The entire UI state is managed through Riverpod providers. The key insight is that the inference state — loading, generating, error — is a stream, not a future. Riverpod's StreamProvider handles this elegantly, giving the UI a reactive view of the token stream without any manual state wiring.

Dart Riverpod provider — chat state 
// Code-generated provider (Riverpod @riverpod annotation)
@riverpod
class ChatNotifier extends _$ChatNotifier {
  final _messages = <ChatMessage[];

  Future<void> sendMessage(String text) async {
    _messages.add(ChatMessage.user(text));
    state = AsyncData(_messages); // triggers immediate UI rebuild

    final buffer = StringBuffer();
    final llm = ref.read(llmServiceProvider);

    await llm.generateStream(
      prompt: text,
      settings: ref.read(settingsProvider).requireValue,
      onToken: (token) {
        buffer.write(token);
        // Update last message in-place — each token triggers rebuild
        _messages.last = ChatMessage.assistant(buffer.toString());
        state = AsyncData([..._messages]);
      },
    );
  }
}

Why Riverpod over BLoC or Provider?

Riverpod's compile-time safety and code generation eliminate an entire class of runtime errors. For a project with this many async service dependencies (LLM engine, TTS, voice input, download manager), the ref.watch / ref.read pattern scales much cleaner than manual stream subscriptions or BLoC events.

How GGUF models work

GGUF (GPT-Generated Unified Format) is the file format that makes on-device inference practical. It's a self-contained binary format that packs model weights, architecture metadata, tokenizer vocabulary, and quantization parameters into a single file — no separate config files, no Python runtime needed.

What's inside a GGUF file

A GGUF file has a structured binary layout. The header section is small and fast to read — this is how ModelMetadataExtractor can show you model info without loading the full weights.

What quantization actually does

A standard LLM stores weights as 16-bit or 32-bit floating point numbers. Quantization reduces each weight to fewer bits — typically 4 bits (Q4) in BRAINY.AI's model catalog. Here's what that means in practice:

Llama 3 8B — size comparison
float32 (original)~32 GB
float16 (half)~16 GB
Q8 (8-bit quant)~8.5 GB
Q4_K_M (4-bit)~4.5 GB
Q2_K (2-bit)~2.9 GB
Quality vs. compression tradeoff
float16 quality100% baseline
Q8 quality~99.5%
Q4_K_M quality~97–98%
Q4_0 quality~95–96%
Q2_K quality~88–92%

The Q4_K_M variant is the sweet spot BRAINY.AI's catalog targets. The "K" means it uses a per-block quantization scheme (k-quants) that's significantly more accurate than naive Q4, and the "M" means medium block size — a balance between speed and accuracy. For most conversational tasks, Q4_K_M output is indistinguishable from float16.

Why memory-mapped access matters

GGUF's 32-byte alignment lets llama.cpp use mmap() to map the weight file directly into virtual address space. This means the OS loads only the tensor pages actually needed for each forward pass — dramatically reducing startup time and RAM overhead compared to loading the full model into heap memory.

How llama.cpp inference works on Android

llama.cpp compiles to native .so shared libraries via the Android NDK and CMake. BRAINY.AI uses the llamadart bindings to call into these libraries from Dart via FFI (Foreign Function Interface). This is the full chain:

Dart (UI thread) Native thread ──────────────── ───────────────────────────────── LLMService.generateStream() │ │ Dart FFI call ▼ llamadart bindings libllama.so (llama.cpp compiled) │ │ │ JNI bridge │ llama_eval() — one forward pass │ │ │ ▼ │ Vulkan / OpenCL kernel dispatch │ │ │ ▼ │ GPU: matrix multiply (GEMM) │ attention scores │ softmax + sampling │ │ │ token callback │ returns token ID ◄──────────────────────────────┘ │ ▼ Dart: onToken(token) → UI rebuild

The KV cache — why context matters

For each forward pass, the transformer computes attention over all previous tokens. Recomputing this from scratch every token would be prohibitively slow. The KV cache stores the key and value matrices from previous tokens so only the new token's attention needs to be computed. This is why context size directly impacts RAM usage — a larger context window means a larger KV cache.

Formula KV cache RAM usage estimate 
// KV cache size (bytes) =
//   2 (K and V)
//   × n_layers
//   × n_heads
//   × head_dim
//   × context_length
//   × bytes_per_element

// Example: Llama 3 8B, 2048 context, Q4
kv_cache = 2 × 32 × 32 × 128 × 2048 × 2 = ~536 MB

// Doubling context to 4096 doubles the KV cache:
// 2 × 32 × 32 × 128 × 4096 × 2 = ~1 GB extra RAM

This is why BRAINY.AI's Low RAM Mode reduces the default context window — it directly cuts KV cache size. The Aggressive Memory Saving setting also forces garbage collection of the KV cache between generations to free memory for other processes.

Live Mode — how continuous conversation works

Live Mode is the more complex feature. It implements a state machine that cycles through four states automatically:

Live Mode State Machine ──────────────────────────────────────────────────── ┌──────────────────────────────────────────┐ │ IDLE │ │ Waiting for user to start speaking │ └──────────────┬───────────────────────────┘ │ audio level > threshold ┌──────────────▼───────────────────────────┐ │ LISTENING │ │ Streaming STT, showing transcript │ │ Visualizer: animated waveform │ └──────────────┬───────────────────────────┘ │ silence detected for 2.5s ┌──────────────▼───────────────────────────┐ │ THINKING │ │ Prompt sent to LLMService │ │ Visualizer: pulsing indicator │ └──────────────┬───────────────────────────┘ │ first token received ┌──────────────▼───────────────────────────┐ │ SPEAKING │ │ TTS reads tokens as they arrive │ │ Visualizer: TTS waveform animation │ └──────────────┬───────────────────────────┘ │ TTS complete → loop back ┌──────────────▼───────────────────────────┐ │ IDLE │ └──────────────────────────────────────────┘ Tap anywhere → interrupt AI mid-response → back to LISTENING
Dart VoiceInputService — silence detection loop 
class VoiceInputService {
  static const _silenceThreshold = Duration(milliseconds: 2500);
  DateTime? _lastSoundAt;

  void _onSoundLevelChanged(double level) {
    if (level > -40.0) {
      _lastSoundAt = DateTime.now(); // reset silence timer
      return;
    }
    final silent = DateTime.now().difference(_lastSoundAt ?? DateTime.now());
    if (silent > _silenceThreshold) {
      _autoSend(); // silence detected — send the accumulated transcript
    }
  }

  Future<void> startListening({
    required Function(String) onPartial,
    required Function(String) onFinal,
  }) async {
    await _stt.listen(
      onResult: (result) => result.finalResult
          ? onFinal(result.recognizedWords)
          : onPartial(result.recognizedWords),
      listenMode: ListenMode.dictation,
      partialResults: true,
      onSoundLevelChange: _onSoundLevelChanged,
    );
  }
}

Text-to-speech — reading responses aloud

The TTSService wraps flutter_tts to read AI responses aloud as they stream in. In Live Mode, TTS is fed tokens as they arrive — it doesn't wait for the full response. This creates a natural spoken cadence rather than a long pause followed by rapid speech.

Technical challenge: TTS + streaming

Feeding tokens one-by-one to TTS sounds choppy. The solution is a sentence accumulator — the service buffers tokens until it detects a sentence boundary (period, question mark, exclamation), then queues the full sentence for TTS. This gives natural pacing while keeping latency low.

💊
Health & medical queries

Asking about symptoms, medications, or mental health without sending sensitive data to any server. Completely private, no retention.

📓
Private journaling

Using the AI as a journaling assistant or reflection partner. Diary entries never leave the device.

💻

Code review, explanation, and generation on planes, in basements, or anywhere without internet. StarCoder2 and CodeQwen handle most tasks.

🏢
Enterprise / sensitive work

Drafting documents, summarising meeting notes, or brainstorming when company policy prohibits sending data to cloud AI services.

🌍
Translation without connectivity

100+ language translation via MADLAD-400 and NLLB-200. Works in areas with no signal — travel, fieldwork, remote locations.

Explaining concepts, solving maths problems (WizardMath), quizzing on material. Runs during lectures where internet use is restricted.

✍️

Story generation, worldbuilding, character dialogue. Mistral 7B in creative personality mode handles long-form creative work well.

🔒
High-security environments

Government, defence, and healthcare contexts where cloud AI is simply not permitted. The app makes zero network requests in offline mode.

🎙️
Voice assistant replacement

Live Mode as an always-available private voice assistant — without sending your voice to Amazon, Google, or Apple.

10

Honest limitations

Hard limit
RAM is an absolute ceiling

If the model + KV cache don't fit in available RAM, the load will fail. There's no graceful degradation. On a device with 6GB total RAM, you realistically have 3–4GB free for the model after OS and other apps. That rules out 7B models entirely on most mid-range devices.

Hard limit
Android 10+ only

The NDK target, Vulkan API level requirements, and biometric API all require API 29+. No workaround planned — the performance APIs on older Android simply aren't there.

Friction
Significant speed variance between devices

The gap between a Snapdragon 8 Gen 3 (50+ t/s) and a budget CPU-only device (8 t/s) is enormous. The same model produces a very different user experience on different hardware. Setting realistic expectations is hard to do in a generic UI.

Friction
Q4 quantization loses accuracy

Especially noticeable on complex reasoning, multi-step maths, and factual recall. Q4_K_M mitigates this significantly vs Q4_0, but it's still not float16. For tasks where exact precision matters, the cloud fallback is worth using.

Friction
Battery drain and thermal throttling

20–30 minutes of sustained inference will noticeably drain battery and cause modern Android thermal managers to throttle the GPU, reducing tokens/sec over time. The Terminate on Background setting helps — it unloads the model when you switch apps.

Minor
Voice input requires internet on some devices

Android's speech recognition engine is on-device for most modern phones, but some manufacturers route STT through the cloud. There's no reliable way to detect or override this at the app level — it's a device/ROM decision.

Minor
No persistent memory across sessions

Each new session starts with an empty context. There's no long-term memory or user profile — RAG support is planned to address this partially, but it's not live yet.

Roadmap
iOS, macOS, Windows — not yet

The Flutter codebase is cross-platform. The blocker is the platform-specific llama.cpp native library compilation and permission models. iOS Metal backend is the next target.
